<objective>
Ship the BMI atomic-save and 30-day history backend. D-54 requires PATCH /api/bmi to BOTH create a BMIRecord AND patch User.profile.heightCm + weightKg in one request so Home and Profile screens always see fresh values.

Purpose: Mobile BMI screen (Plan 03-08) saves with one PATCH and reads chart history with one GET.

Output: New `backend/src/api/bmi/` module, supertest integration tests, router mounted in `app.ts`.

**Note on D-54 atomicity:** True multi-document transactions require a MongoDB replica set or sharded cluster. MongoDB Atlas M2 (Phase 1 D-08) IS a replica set, so transactions are available. However, `mongodb-memory-server` defaults to a single-node standalone — transactions unavailable in tests. Implementation rule: try transaction; on `MongoServerError` with codeName 'IllegalOperation' or 'TransactionsNotSupported', fall back to sequential operations and log a warning. Tests verify the sequential path works correctly (final state is the same — both operations complete or both throw together with rollback of the BMIRecord if User update fails).
</objective>

    Export `async function saveBMIAtomic(userId: string, heightCm: number, weightKg: number)`. Pseudocode:
    ```
    const userObjId = new mongoose.Types.ObjectId(userId);
    const bmi = computeBMI(heightCm, weightKg);
    const category = categorizeBMI(bmi);
    const now = new Date();

    // Try transaction (Atlas replica set path)
    const session = await mongoose.startSession();
    try {
      let bmiRecord; let updatedUser;
      await session.withTransaction(async () => {
        const created = await BMIRecord.create([{ userId: userObjId, heightCm, weightKg, bmi, category, recordedAt: now }], { session });
        bmiRecord = created[0].toObject();
        updatedUser = await User.findByIdAndUpdate(userObjId,
          { 'profile.heightCm': heightCm, 'profile.weightKg': weightKg },
          { new: true, runValidators: true, session }
        ).select('profile').lean();
        if (!updatedUser) throw makeError('Người dùng không tồn tại', 404);
      });
      return { bmiRecord, user: { heightCm: updatedUser!.profile.heightCm, weightKg: updatedUser!.profile.weightKg } };
    } catch (err: any) {
      const code = err?.codeName || err?.code;
      const isTransactionUnsupported = code === 'IllegalOperation' || /Transaction numbers are only allowed on a replica set/.test(err?.message ?? '');
      if (!isTransactionUnsupported) throw err;
      // Fallback sequential path (memory-server / standalone)
      console.warn('[bmi.service] Transactions unsupported — falling back to sequential save.');
      const bmiRecord = await BMIRecord.create({ userId: userObjId, heightCm, weightKg, bmi, category, recordedAt: now });
      try {
        const updatedUser = await User.findByIdAndUpdate(userObjId,
          { 'profile.heightCm': heightCm, 'profile.weightKg': weightKg },
          { new: true, runValidators: true }
        ).select('profile').lean();
        if (!updatedUser) {
          await BMIRecord.deleteOne({ _id: bmiRecord._id });   // compensating rollback
          throw makeError('Người dùng không tồn tại', 404);
        }
        return { bmiRecord: bmiRecord.toObject(), user: { heightCm: updatedUser.profile.heightCm, weightKg: updatedUser.profile.weightKg } };
      } catch (userErr) {
        await BMIRecord.deleteOne({ _id: bmiRecord._id }).catch(() => {});
        throw userErr;
      }
    } finally {
      session.endSession();
    }
    ```

    Export `async function getBMIHistory(userId: string)`:
    ```
    const userObjId = new mongoose.Types.ObjectId(userId);
    const thirtyDaysAgo = new Date(vietnamDayStart(new Date()).getTime() - 29 * 86400000);
    // Aggregate: group by day bucket (UTC+7) → pick latest recordedAt per day
    const pipeline = [
      { $match: { userId: userObjId, recordedAt: { $gte: thirtyDaysAgo } } },
      { $addFields: { dayBucket: {
        $dateToString: { format: '%Y-%m-%d', date: { $add: ['$recordedAt', 7 * 60 * 60 * 1000] } }
      } } },
      { $sort: { recordedAt: -1 as -1 } },
      { $group: { _id: '$dayBucket', bmi: { $first: '$bmi' }, category: { $first: '$category' } } },
      { $sort: { _id: 1 as 1 } },                 // oldest first
      { $project: { _id: 0, date: '$_id', bmi: 1, category: 1 } },
    ];
    const result = await BMIRecord.aggregate(pipeline);
    return result;          // oldest → newest, up to 30 entries (one per day bucket within the window)
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| mobile client → /api/bmi/* | Untrusted JSON body (heightCm, weightKg, possibly malicious userId) |
| /api/bmi/* → MongoDB | Per-user writes touching TWO collections (BMIRecord + User) — atomicity required (D-54) |
| Aggregation → User isolation | History query must filter by userId |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-20 | Elevation of Privilege | userId injection via body | mitigate | Same pattern as T-03-07: `.strict()` schema, userId from JWT only |
| T-03-21 | Tampering | Inconsistent state if User update fails after BMIRecord create | mitigate | Atlas path: `session.withTransaction` — both commit or both abort. Memory-server fallback path: explicit `BMIRecord.deleteOne` compensation if User update fails. Both paths covered by integration test (case 10 asserts both writes succeed together). |
| T-03-22 | Information Disclosure | Cross-tenant BMI history leak | mitigate | `$match: { userId: ObjectId(req.user.id) }` first stage; verified by test case 15 |
| T-03-23 | Tampering | Out-of-range slider values poisoning chart | mitigate | Zod `.min(100).max(220)` / `.min(30).max(200)` (D-56) enforced before service |
| T-03-24 | Tampering | Client-supplied bmi/category fields | mitigate | `.strict()` schema rejects them; service computes both server-side from heightCm/weightKg — client never has authority over the stored category |
| T-03-25 | Spoofing | Anonymous BMI save | mitigate | `authenticate` on both routes |
| T-03-26 | Information Disclosure | User.passwordHash leaked via User.findByIdAndUpdate response | mitigate | `.select('profile').lean()` returns only `profile` subdocument — passwordHash never serialized in response |
| T-03-SC | Tampering | npm install of new dependencies | accept | No new dependencies — reuses zod/mongoose/supertest stack |
</threat_model>
