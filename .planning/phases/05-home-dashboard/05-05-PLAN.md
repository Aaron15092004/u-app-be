This is a mobile plan. Phase 5 mobile plans (05, 06, 07) intentionally do NOT include React Native Testing Library behavioral tests in their `<verify>` blocks. The automated verification is `npx tsc --noEmit` (TypeScript strict compilation).

**Why behavioral tests are deferred for mobile in Phase 5:**
- Per STATE.md: "MMKV, Google Sign-In, Apple Sign-In mocked for Expo Go testing" — the project's existing mobile test infrastructure relies on Expo Go and does NOT have a CI-runnable harness for the native modules this phase exercises (`@react-native-community/datetimepicker`, `expo-linear-gradient`, `expo-notifications`, MMKV).
- `expo-notifications` permission flow, FCM token retrieval, and DateTimePicker UI cannot be exercised in jsdom or Expo Go — they require an EAS development build on a real device.
- TypeScript strict compilation catches API contract drift between mobile types (Plan 05) and backend response shapes (Plans 02, 03, 04). This is the primary regression risk for an interface-first plan.

**End-to-end behavioral verification:**
All Phase 5 mobile behavioral verification is consolidated into the blocking-human checkpoint at **Plan 07 Task 3** ("Phase 5 device verification"). That checkpoint executes 15 manual steps on a real device covering: home dashboard render, water log +/- flow, profile tab navigation, edit profile PATCH, notification settings, rationale modal, FCM water/workout reminders, FCM streak alert at 20:00 UTC+7, help screen, logout. This satisfies the Nyquist behavioral coverage requirement at the phase level even though individual mobile tasks rely on tsc only.

- Backend plans (01–04, Waves 0–1): full integration test suites via `npm run test:home`, `npm run test:water`, `npm run test:users`, plus `npx tsc --noEmit`
- Mobile plans (05–07, Waves 2–3): `npx tsc --noEmit` per task; full E2E behavioral verification at Plan 07 Task 3 device checkpoint
