## Objective

Build the core Express application layer: middleware stack (cors, helmet, express.json, rate-limit), the Mongoose loader with Atlas M2 connection settings, seven Mongoose models with all required compound indexes (per D-11), and the `/api/health` endpoint that proves the DB connection is truly live by writing, reading, and deleting a test document via Mongoose. This plan delivers Walking Skeleton success criteria 2 and 3.

A critical constraint from PITFALLS.md: the Mongoose connection MUST be initialized with `maxPoolSize: 10` and `serverSelectionTimeoutMS: 5000` to prevent Atlas M0/M2 connection exhaustion under load. The compound indexes on `{ userId: 1, date: -1 }` MUST be created in this phase (per D-11) — not deferred to Phase 3/4.

**CHK-01 fix:** The health controller performs a real write/read/delete round-trip on a `HealthCheckLog` collection (a dedicated lightweight collection). This detects Atlas IP whitelist denials, wrong database name, and permission errors that a `readyState` check alone cannot surface.

<action>
Create `backend/src/api/health/health.routes.ts` — import Router from express, import healthController. Register: router.get('/', healthController.check). Export router. This mounts at /api/health (the prefix is set in app.ts).

Create `backend/src/api/health/health.controller.ts` — export async function check(req: Request, res: Response): Promise<void>. The handler performs the following steps:

Step 1 — DB write/read/delete round-trip: import HealthCheckLog model. Create a new HealthCheckLog document with { checkedAt: new Date(), host: 'health' }, save it (await doc.save()), read it back (await HealthCheckLog.findById(doc._id)), then delete it (await HealthCheckLog.deleteOne({ _id: doc._id })). If all three operations succeed without throwing, set dbWrite = true. Wrap the entire round-trip in a try/catch: if any operation throws, set dbWrite = false and log the error with console.error('Health DB write check failed:', err). This detects Atlas IP whitelist denials, wrong database name, or permission errors that readyState alone cannot surface.

Step 2 — Build response: return success(res, { status: 'ok', db: mongoose.connection.readyState === 1 ? 'connected' : 'disconnected', dbWrite, version: process.env.npm_package_version ?? '1.0.0', environment: config.NODE_ENV, timestamp: new Date().toISOString() }). The response always returns HTTP 200 for the health check (degraded state is reflected in the dbWrite: false field — the endpoint itself must remain reachable for monitoring probes). If mongoose.connection.readyState !== 1 AND dbWrite is false, return error(res, 'Database not connected', 503) instead.

This endpoint must NOT require authentication — it is a public health probe used by Render and the Walking Skeleton test.
</action>

| Threat | Severity | Mitigation |

| MONGODB_URI with credentials in plain .env committed to git | HIGH | .env is in .gitignore (Plan 01); .env.example contains only placeholder; config/index.ts reads from process.env at runtime only |

| MongoDB injection via unvalidated query inputs | MEDIUM | Mongoose schema type enforcement prevents most injection; Zod validation middleware is a Phase 2 addition for all user inputs |
| Mongoose connection left open on process exit | LOW | error.middleware.ts + unhandledRejection handler calls process.exit(1); Mongoose closes connections on process exit by default |
| HealthCheckLog collection accumulating stale documents on crash | LOW | Each health check deletes its own document; on crash before delete, document stays. Collection is ephemeral and can be dropped/TTL-indexed without data loss. Add TTL index: HealthCheckLogSchema.index({ checkedAt: 1 }, { expireAfterSeconds: 300 }) so Atlas auto-expires documents older than 5 minutes |
