Per D-12: Cloudinary upload flows through the backend (Multer → Cloudinary SDK). Per D-13: Firebase Admin SDK initialized server-side. Per D-14: LogMeal + GPT-4o-mini clients exist as stubs — no implementation in Phase 1, only configuration.

| Threat | Severity | Mitigation |
|--------|----------|------------|
| FIREBASE_PRIVATE_KEY exposed in environment variable | HIGH | Private key is a runtime env var only; .env is gitignored; on Render, set via dashboard env vars — never commit to git or log to console |
| CLOUDINARY_API_SECRET exposed in logs or error messages | HIGH | cloudinary.service.ts does not log config values; error handler in error.middleware.ts does not include request body or env vars in error responses |
| Test upload endpoint accessible in production (information disclosure) | HIGH | upload-test.routes.ts has TODO comment to remove before production; Plan 05 CI/CD should add a grep check for this route being absent in production builds |
| Unauthenticated push token registration allows spam token injection | MEDIUM | Phase 1: accepted as known temporary gap (auth not implemented yet); Phase 2 MUST add JWT auth middleware to this route — enforced by TODO comment |
| Firebase Admin SDK re-initialization on hot reload crashing server | LOW | firebase.ts checks admin.apps.length > 0 before calling initializeApp(); this prevents 'Firebase App named default already exists' error during tsx watch restarts |
| Multer memory storage causing OOM on large file upload | MEDIUM | 5MB file size limit enforced in upload.middleware.ts; additional defense: image compression on mobile side (D-24, <500KB); server-side limit is defense-in-depth |
</threat_model>
