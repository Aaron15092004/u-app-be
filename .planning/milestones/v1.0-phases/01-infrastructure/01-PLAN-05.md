<action>
Create `mobile/eas.json` — EAS build configuration with exactly 3 profiles as decided in CONTEXT.md.

Profile 1 — `development`: { "developmentClient": true, "distribution": "internal", "android": { "buildType": "apk" }, "ios": { "simulator": false } }. The developmentClient: true flag creates a Development Build (not Expo Go). distribution: internal means shared via EAS internal distribution URL. Android: APK format (faster install for dev testing). iOS: real device (not simulator — simulator builds need Xcode locally).

Profile 2 — `preview`: { "distribution": "internal", "android": { "buildType": "apk" }, "ios": { "simulator": false } }. No developmentClient — this is a production-like build for internal testing. Used for stakeholder demos and pre-App Store testing.

Profile 3 — `production`: { "distribution": "store", "android": { "buildType": "app-bundle" }, "ios": { "simulator": false } }. distribution: store means the output is submitted to App Store / Play Store. Android: AAB format required for Play Store. iOS: default IPA.

Global settings: { "cli": { "version": ">= 14.0.0" }, "build": { ... the three profiles above ... }, "submit": { "production": {} } }.

Create `mobile/.env.example` — EXPO_PUBLIC_API_URL=https://u-app-backend.onrender.com. Note: This is the Render deployment URL (to be updated after Render deploy is configured in Task 3). For local development, override with: EXPO_PUBLIC_API_URL=http://localhost:3000 (never commit a .env file with the localhost override).

Document the build commands in a comment at the top of eas.json (as a README note — JSON does not support comments, so create `mobile/EAS-README.md`): Development build: `eas build --profile development --platform ios` and `eas build --profile development --platform android`. First-time setup requires: `eas init` to link to EAS project, `eas credentials` to configure signing. EAS Free tier: 100 builds/month shared queue (~15-30 min wait time).
</action>

<threat_model>
| Threat | Severity | Mitigation |
|--------|----------|------------|
| Secrets leaked via render.yaml (sync: false bypassed) | HIGH | All render.yaml envVar entries have sync: false; no actual secret values in the YAML file; accepted by grep check in acceptance criteria |
| CI/CD pipeline running with elevated cloud credentials | MEDIUM | CI workflow only runs typecheck and lint — no deploy, no cloud SDK calls, no credentials required; Render auto-deploy is triggered via Render's GitHub integration webhook, not via CI credentials |
| .env file accidentally committed despite .gitignore | HIGH | .gitignore at root and in each workspace covers .env and *.env.local; pre-commit hook (optional, add in Phase 2+) can check for .env files |
| EAS project ID in app.json being a placeholder | LOW | extra.eas.projectId is set to REPLACE_WITH_EAS_PROJECT_ID placeholder; eas build will prompt for the real ID on first run; this is intentional for initial scaffold |
| render.yaml region choice (singapore) exposing data to specific jurisdiction | LOW | Singapore is closest low-latency region to Vietnam on Render's available regions; data residency requirements for Vietnamese health data are not currently regulated; document this decision in the render.yaml comment |
| Node.js version mismatch between CI (20) and Render (runtime) | LOW | render.yaml does not specify Node version; Render defaults to active LTS; backend/package.json engines field should be added: { "node": ">=20.0.0" } to enforce version |
</threat_model>
