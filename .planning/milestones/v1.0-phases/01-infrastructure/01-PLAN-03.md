## Objective

Build the mobile application's foundation layer: the Expo Router route groups structure, the three providers (QueryProvider using TanStack Query v5, AuthProvider as a stub for Phase 2, ThemeProvider with color tokens), the Axios API client with base configuration and interceptor placeholders, a minimal index screen that exercises the end-to-end mobile to backend path by calling `/api/health`, and a push token registration module that calls `getExpoPushTokenAsync` on mount and POSTs the token to the backend.

This plan runs in Wave 2 (parallel to Plan 02) because it only depends on the scaffold created in Plan 01. The health-check screen will work once Plan 02's backend is running.

Key constraints from PITFALLS.md and CONTEXT.md decisions:
- D-16: Zustand + TanStack Query (no Redux)
- D-17: expo-secure-store for JWT storage (never AsyncStorage)
- D-22: Access token in memory (Zustand), refresh token in expo-secure-store
- Walking Skeleton SC-5: Expo push token can be registered from the device

<action>
Create `mobile/src/lib/notifications/push-token.ts` — This module handles Expo push token registration. It calls getExpoPushTokenAsync, then POSTs the token to the backend so it can be stored in MongoDB and used for FCM delivery.

Import * as Notifications from 'expo-notifications'. Import Constants from 'expo-constants'. Import the Axios apiClient from '../api/client'.

Export async function registerPushToken(apiClient: AxiosInstance): Promise<{ success: boolean; token?: string; error?: string }>. Implementation:

Step 1 — Get Expo push token: call await Notifications.getExpoPushTokenAsync({ projectId: Constants.expoConfig?.extra?.eas?.projectId }). The projectId is required for physical device token generation; it reads from app.json extra.eas.projectId. Store the result as expoPushToken.data (the token string in format 'ExponentPushToken[xxxxxx]').

Step 2 — POST to backend: call await apiClient.post('/api/notifications/register-token', { token: expoPushToken.data, platform: Platform.OS === 'ios' ? 'ios' : 'android' }). Import Platform from 'react-native'.

Step 3 — Return { success: true, token: expoPushToken.data } on success.

Wrap the entire function body in a try/catch. On any error (permissions denied, network failure, invalid projectId, backend unavailable): log console.warn('Push token registration failed (non-blocking):', err.message), return { success: false, error: err.message }. Push token registration MUST be non-blocking — failure here must never prevent the app from loading or displaying the health-check screen. The return value lets callers display registration status without crashing.

Note: On Android, request notification permissions before calling getExpoPushTokenAsync. Add: const { status } = await Notifications.requestPermissionsAsync(); if (status !== 'granted') return { success: false, error: 'Notification permission denied' };

Export the function as a named export (not default). The caller (index.tsx) will import it and call it on mount.

<threat_model>
| Threat | Severity | Mitigation |
|--------|----------|------------|
| JWT refresh token stored in AsyncStorage (insecure on Android) | HIGH | token-storage.ts exclusively uses expo-secure-store; AsyncStorage import is forbidden in this file; acceptance criteria checks for absence of AsyncStorage |
| Access token logged to console or included in error reports | MEDIUM | Axios interceptor in client.ts does not log token values; error handler re-throws without token data; Sentry/analytics integration (Phase 2+) must filter Authorization headers |
| EXPO_PUBLIC_ env var exposed in bundle (by design for public config) | LOW | EXPO_PUBLIC_API_URL is the backend URL, not a secret; acceptable to be in bundle — real secrets (API keys) must use server-side env vars only |
| TanStack Query cache containing auth data persisted to disk | LOW | Phase 1 QueryClient has no persister configured; auth state is in AuthProvider memory only |
| CORS misconfiguration allowing mobile to call wrong backend | LOW | EXPO_PUBLIC_API_URL defaults to localhost:3000; production URL must be set as EAS env var before building production profile |
| Push token registration crashing app on permission denied | MEDIUM | registerPushToken() wraps all operations in try/catch and returns { success: false } — never throws; permission denial is handled as a non-fatal failure |
| Invalid EAS projectId causing push token fetch to fail silently | LOW | Failure is caught and logged as a warning; app continues loading; SC-5 can be re-verified once eas init assigns real projectId |
</threat_model>
