  Request: { email: string; password: string }
  Success 200: { success: true, data: { user: SafeUser, accessToken: string, refreshToken: string } }
  Errors: 400 (validation), 401 ("Email hoặc mật khẩu không đúng")

POST /api/auth/refresh
  Request: { refreshToken: string }
  Success 200: { success: true, data: { accessToken: string, refreshToken: string } }   // rotated
  Errors: 400 (validation), 401 (invalid/expired/revoked — "Phiên đăng nhập hết hạn. Vui lòng đăng nhập lại." per D-39)

POST /api/auth/logout
  Auth: Bearer access token required
  Request: {}  (no body)
  Success 204
  Errors: 401 (no/invalid token)

PATCH /api/auth/complete-profile
  Auth: Bearer access token required
  Request: { name: string (non-empty); age: number (10..120); heightCm: number (50..300); weightKg: number (20..500); goalType: 'lose'|'maintain'|'gain' }   // per D-31
  Success 200: { success: true, data: { user: SafeUser } }   // user.profileCompleted = true
  Errors: 400 (validation), 401 (no token)

    `registerWithEmail(email: string, password: string)`:
    - Lowercase + trim email.
    - Check existing: `User.findOne({ email })` — if found, throw error with `statusCode=409` and message "Email này đã được sử dụng. Đăng nhập hoặc dùng email khác".
    - Hash password via `hashPassword`.
    - Create user: `User.create({ email, passwordHash, profileCompleted: false })`. Other fields use schema defaults (name=''; authProviders=[]; etc).
    - Generate token pair via private helper `issueTokenPair(user)` (see below).
    - Return `{ user: toSafeUser(user), accessToken, refreshToken }`.

    `loginWithEmail(email: string, password: string)`:
    - Lowercase email; `User.findOne({ email })`.
    - If user missing OR `!user.passwordHash` (OAuth-only user) OR `comparePassword(password, user.passwordHash) === false` → throw `statusCode=401`, message "Email hoặc mật khẩu không đúng".
    - Return `issueTokenPair(user)` + SafeUser.

    `rotateRefreshToken(rawRefreshToken: string)`:
    - `verifyRefreshToken(rawRefreshToken)` → `{ sub, jti }`. If signature fails, throw `statusCode=401`, message "Phiên đăng nhập hết hạn. Vui lòng đăng nhập lại." (D-39).
    - `User.findById(sub)` — if missing, throw 401.
    - If `!user.refreshTokenHash` OR `compareTokenHash(rawRefreshToken, user.refreshTokenHash) === false` → throw 401 (token reuse / revoked).
    - If `user.refreshTokenExpiry < new Date()` → throw 401 (D-22 7-day cap).
    - Generate new pair via `issueTokenPair(user)` — this writes a new hash, invalidating the old raw token (D-22 rotate-on-use).
    - Return `{ accessToken, refreshToken }`.

    `revokeRefreshToken(userId: string)`:
    - `User.findByIdAndUpdate(userId, { refreshTokenHash: null, refreshTokenExpiry: null })`.

    `updateProfile(userId: string, payload: CompleteProfileInput)`:
    - Derive dateOfBirth from age: `const dateOfBirth = new Date(); dateOfBirth.setFullYear(dateOfBirth.getFullYear() - payload.age);`. Storing only `age` is rejected per D-31 (the User schema has `dateOfBirth`); we approximate as Jan-1 of birth year to keep DOB usable without asking for exact date. NOTE in summary: this is an MVP simplification — could be replaced with a real date picker in v2.
    - `User.findByIdAndUpdate(userId, { name: payload.name, 'profile.dateOfBirth': dateOfBirth, 'profile.heightCm': payload.heightCm, 'profile.weightKg': payload.weightKg, 'profile.goalType': payload.goalType, profileCompleted: true }, { new: true, runValidators: true })`.
    - If null → throw `statusCode=404`, message "Người dùng không tồn tại".
    - Return SafeUser.

    Private helpers in same file:
    ```typescript
    function toSafeUser(user: IUser): SafeUser {
      return { id: user._id.toString(), email: user.email, name: user.name, role: user.role, profileCompleted: user.profileCompleted };
    }
    async function issueTokenPair(user: IUser): Promise&lt;{ accessToken: string; refreshToken: string }&gt; {
      const jti = crypto.randomUUID();
      const accessToken = signAccessToken({ sub: user._id.toString(), role: user.role });
      const refreshToken = signRefreshToken({ sub: user._id.toString(), jti });
      const refreshTokenHash = await hashToken(refreshToken);
      const expiryMs = 7 * 24 * 60 * 60 * 1000;
      await User.findByIdAndUpdate(user._id, { refreshTokenHash, refreshTokenExpiry: new Date(Date.now() + expiryMs) });
      return { accessToken, refreshToken };
    }
    ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| mobile client → /api/auth/* | Untrusted JSON bodies cross here — must zod-validate before service layer |
| /api/auth/* service → MongoDB | Password and refresh token hashes written |
| /api/auth/refresh | Token rotation must invalidate old token to prevent replay |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-14 | Tampering | request body | mitigate | zod `safeParse` at controller; reject 400 with Vietnamese error before service touches data |
| T-02-15 | Spoofing | refresh token replay after rotation | mitigate | `rotateRefreshToken` writes new hash; old raw token fails `compareTokenHash` (D-22 rotate-on-use) |
| T-02-16 | Information Disclosure | password in error message | mitigate | controller returns generic "Email hoặc mật khẩu không đúng" — never leaks whether email exists vs password wrong |
| T-02-17 | Brute force password guessing | login endpoint | accept (Phase 2) | global rate limiter (100 req / 15 min) from Phase 1 loadExpress() applies; per-account lockout deferred to v2 |
| T-02-18 | Information Disclosure | response includes refreshTokenHash | mitigate | `toSafeUser()` whitelists fields; assertion in integration test that response.body.data.user has no `refreshTokenHash` or `passwordHash` keys |
| T-02-19 | Elevation of Privilege | unauthenticated logout/complete-profile | mitigate | `authenticate` middleware mounted on those routes |
| T-02-20 | Tampering | password too short bypasses rule | mitigate | zod `password: z.string().min(8)` enforces AUTH-02 "min 8 chars" rule |
| T-02-SC | Tampering | new npm dependencies (supertest, mongodb-memory-server) | mitigate | Both are widely used, well-maintained (supertest >1M weekly DL; mongodb-memory-server official testing toolkit). Verified legitimate during planning; devDependencies only — not shipped to production. |
