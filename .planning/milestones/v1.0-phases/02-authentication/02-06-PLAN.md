    `consumePasswordReset(rawToken: string, newPassword: string)`:
    - There's no index on passwordResetTokenHash (it's bcrypt — non-deterministic). Strategy: query all users with `passwordResetTokenExpiry > new Date()` and `passwordResetTokenHash != null`. For typical usage at MVP scale this is small (<100). For each candidate, run `compareTokenHash(rawToken, user.passwordResetTokenHash)` and break on first match.
    - This works but is O(n) — acceptable for MVP. Note in summary: "scan strategy chosen because bcrypt hashes are not directly queryable; replace with HMAC-SHA256 or random-prefix index strategy in v2 if scale demands."
    - If no match found → throw `statusCode=401`, message "Link đặt lại mật khẩu đã hết hạn. Vui lòng yêu cầu lại." (UI-SPEC system messages).
    - On match: update user with `passwordHash: await hashPassword(newPassword), passwordResetTokenHash: null, passwordResetTokenExpiry: null, refreshTokenHash: null, refreshTokenExpiry: null` (revokes all sessions per security best practice).
    - Return `{ message: 'Đặt lại mật khẩu thành công' }`.

    Extend `backend/src/api/auth/auth.controller.ts` with two handlers:

    ```typescript
    export async function forgotPassword(req: Request, res: Response): Promise<void> {
      const parsed = forgotPasswordSchema.safeParse(req.body);
      if (!parsed.success) { error(res, parsed.error.errors[0].message, 400); return; }
      await authService.issuePasswordReset(parsed.data.email);
      // Always return the same response — do not leak whether email exists
      success(res, { message: 'Email đặt lại mật khẩu đã được gửi! Kiểm tra hộp thư của bạn.' });
    }

    export async function resetPassword(req: Request, res: Response): Promise<void> {
      const parsed = resetPasswordSchema.safeParse(req.body);
      if (!parsed.success) { error(res, parsed.error.errors[0].message, 400); return; }
      const result = await authService.consumePasswordReset(parsed.data.token, parsed.data.password);
      success(res, result);
    }
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| POST /api/auth/forgot-password | Untrusted email — must not reveal whether account exists |
| Reset token in email | Raw token in plain text in email body — must be single-use + time-limited |
| Reset token at rest | bcrypt hash only — never the raw |
| Deep link parsing | Token from URL — must be treated as untrusted input |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-28 | Information Disclosure | user enumeration via forgot-password | mitigate | Controller returns identical 200 message regardless of whether email exists; service skips DB write + email send if missing |
| T-02-29 | Tampering | reset token brute force | mitigate | 32 bytes (256 bits) of crypto.randomBytes → infeasible to guess; bcrypt at rest |
| T-02-30 | Replay | reset token reuse | mitigate | passwordResetTokenHash set to null after consumePasswordReset; subsequent attempts return 401 |
| T-02-31 | Replay | expired token still works | mitigate | passwordResetTokenExpiry checked in compareTokenHash loop; expired entries skipped |
| T-02-32 | Spoofing | session hijack after reset | mitigate | consumePasswordReset also sets refreshTokenHash=null, refreshTokenExpiry=null — revokes ALL active sessions (defense in depth) |
| T-02-33 | Tampering | tampered token in deep link | mitigate | Backend validates bcrypt match; tampered/random tokens fail match → 401 |
| T-02-34 | Information Disclosure | email content leaked via Resend logs | accept | Resend stores subject + recipient in their dashboard; no PII in subject; recipient is the user themselves |
| T-02-SC | Tampering | crypto.randomBytes | mitigate | Node built-in; no third-party install required for token generation |
</threat_model>
